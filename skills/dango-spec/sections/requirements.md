# 設計書: 要件定義書

「何を・誰のために・どこまで作るか」を確定する。実装より上流。
日本の現場では機能要件と**非機能要件**を分けて書くのが定石。

## 構成(雛形)

```
# 要件定義書 — {{プロジェクト名}}

## 改訂履歴
| 版 | 日付 | 変更者 | 変更内容 |
|---|---|---|---|
| 1.0 | {{日付}} | {{氏名}} | 初版 |

## 1. 目的・背景
{{解決したい課題と、このシステムで実現したいこと}}

## 2. スコープ
- 対象とする業務/機能: {{ }}
- 対象外(やらないこと): {{ }}

## 3. 用語定義
| 用語 | 意味 |
|---|---|
| {{ }} | {{ }} |

## 4. 機能要件
| ID | 機能名 | 概要 | 優先度 |
|---|---|---|---|
| FR-01 | {{ }} | {{ }} | 高/中/低 |

## 5. 非機能要件
- 性能: {{応答時間・同時接続など}}
- 可用性: {{稼働率・障害時の扱い}}
- セキュリティ: {{認証・権限・個人情報の扱い → dango-legal と接続}}
- 互換性: {{対応ブラウザ・OS・端末}}
- 運用・保守: {{ログ・監視・バックアップ}}

## 6. 制約・前提
{{予算・期日・既存システム・法令(特商法/個情法など)}}

## 7. 受け入れ条件
{{満たすべき観察可能な状態}}
```

## 原則
- 機能要件にIDを振り、後段の設計・テストから追跡できるようにする。
- 非機能要件を省略しない(日本の現場でレビュー指摘が集中する箇所)。
- 法令・個人情報が絡むなら `dango-legal` の確認を前提として明記する。

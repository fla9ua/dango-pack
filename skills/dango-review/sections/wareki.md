# 観点: 和暦・元号・年度

日本固有の暦まわり。改元(元号切り替え)で壊れるコードは「いつか必ず動かなく
なる時限爆弾」なので重大度を高めに見る。

## 検出パターン

### 1. 改元に弱い決め打ち
- `令和` や `2019年5月1日` 起点を定数でハードコードして西暦変換している。
- `if (year >= 2019) "令和"` のような分岐は次の改元で破綻する。
- Grep 例: `rg '令和|平成|昭和|元号|wareki|gengo|和暦'`

### 2. 和暦変換は標準APIに寄せる
- 自前テーブルより `Intl.DateTimeFormat` の和暦カレンダーを推奨:
  ```js
  new Intl.DateTimeFormat('ja-JP-u-ca-japanese', { era: 'long', year: 'numeric' })
    .format(date) // → 令和7年
  ```
- 自前変換が必要な理由(帳票要件など)があるか確認し、無ければ標準APIを提案。

### 3. 年度(4月始まり)
- 「年度」は1〜3月が前年度になる。`getFullYear()` をそのまま年度として
  使っていないか。
- 集計・締め・学年・会計期間で年度ロジックが要るのに月を見ていない箇所。
- Grep 例: `rg '年度|fiscal|nendo'`

### 4. 日付フォーマットとロケール
- `toLocaleDateString()` をロケール指定なしで使うと環境依存。`'ja-JP'` を明示。
- `YYYY/MM/DD` と `YYYY-MM-DD` と `YYYY年M月D日` の混在。
- タイムゾーン: サーバUTC / 表示JST(+09:00)のずれ。日付の境界(深夜)で
  1日ずれるバグに注意。

## 直し方の指針
- 和暦表示は `Intl.DateTimeFormat('ja-JP-u-ca-japanese')` を第一候補に。
- 改元定数のハードコードを見たら「次の改元で壊れる」と明示して警告。
- 年度が絡む計算は「4月始まり」を関数に切り出して一箇所に。
